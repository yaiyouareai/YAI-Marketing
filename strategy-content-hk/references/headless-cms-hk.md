# Headless CMS 指南（香港市場版）

為香港市場的行銷內容選擇、建模和實施 Headless CMS 的參考指南。

## 使用時機

當你為新項目選擇 CMS、為行銷網站設計內容模型、設置編輯工作流程，或將 CMS 內容連接到程式化頁面時，使用此參考。

**香港市場特別考量：**
- 雙語內容需求（中文 + 英文）
- 需要支援 WhatsApp 分享預覽（og:image 優化）
- 本地 CDN 覆蓋（Cloudflare 在香港的節點）
- 如有大陸市場需求，需要考慮 ICP 備案和中國 CDN

---

## Headless vs 傳統 CMS

Headless CMS 將內容管理與呈現分離。內容儲存在結構化後端，通過 API 傳遞到任何前端。

### 適合使用 Headless 的情況

- 多個前端使用相同內容（網站、手機應用、電郵）
- 開發者需要對前端技術棧的完全控制
- 內容需要在不同渠道重複使用
- 使用現代框架建置（Next.js、Remix、Astro）
- 行銷需要結構化、可重用的內容塊
- 需要支援雙語（中英文）內容

### 適合使用傳統 CMS 的情況

- 小型團隊沒有專職開發者
- 簡單的部落格或介紹網站
- WYSIWYG 編輯是硬性要求
- 預算有限，WordPress / Webflow 夠用

### 決策清單

| 因素 | Headless | 傳統 |
|------|---------|------|
| 多渠道傳遞 | 是 | 有限 |
| 開發者控制 | 完全 | 受限 |
| 非技術人員編輯 | 需要設置 | 內建 |
| 上線時間 | 較長 | 較快 |
| 內容重用 | 原生支援 | 手動 |
| 雙語支援 | 需要配置 | 視平台而定 |
| 中國大陸訪問 | 需要額外配置 | 視託管而定 |

## 香港行銷內容建模

### 核心原則

1. **以類型思考，而非頁面。** 「登陸頁」是一個帶字段的內容類型——不是 HTML 文件。這讓你可以在頁面之間重用組件。
2. **內容與呈現分離。** 儲存標題文字，而非帶樣式的標題。呈現屬於前端。
3. **為重用而設計。** 如果客戶評語出現在 5 個頁面，建立一個 Testimonial 類型並引用它——不要重複。
4. **保持模型扁平。** 深度嵌套的結構難以查詢和維護。偏好引用而非嵌套。
5. **為雙語設計。** 從一開始就在模型中加入語言字段，避免後期重構。

### 香港行銷常用內容類型

| 類型 | 關鍵字段 | 備注 |
|------|---------|------|
| **登陸頁** | title_en, title_zh, slug, hero, sections[], seo | 加入中英文標題字段 |
| **部落格文章** | title_en, title_zh, body, author, category, tags, publishedAt, seo, language | 支援中英文雙語版本 |
| **案例研究** | title, customer, industry, challenge, solution, results, metrics[], logo | 優先收集本地香港案例 |
| **客戶評語** | quote_en, quote_zh, author, role, company, avatar, rating | 中英文評語分開儲存 |
| **FAQ** | question_en, question_zh, answer_en, answer_zh, category | 適合雙語搜尋優化 |
| **作者** | name, bio_en, bio_zh, avatar, social links | 包含 LinkedIn、小紅書連結 |
| **CTA 區塊** | heading_en, heading_zh, body, buttonText, buttonUrl, variant | 跨頁面可重用 |
| **本地指南** | title, region（香港/大灣區/亞太）, body, seo | 針對本地搜尋的特殊類型 |

### SEO 字段清單（雙語版）

每個頁面級內容類型需要：

- `metaTitle_en` — 50-60 個英文字符
- `metaTitle_zh` — 25-30 個中文字符
- `metaDescription_en` — 150-160 個英文字符
- `metaDescription_zh` — 75-80 個中文字符
- `ogImage` — 1200x630px 社交預覽（對 WhatsApp 分享尤其重要）
- `slug` — URL 路徑片段
- `canonicalUrl` — 可選的正規 URL 覆蓋
- `noIndex` — 布林值，用於從搜尋中排除
- `hreflang` — 中英文版本之間的語言標記

## 編輯工作流程

### 草稿 → 審閱 → 發布循環

1. **草稿** — 作者創建或編輯內容
2. **語言審閱** — 確保中英文版本一致
3. **編輯審閱** — 編輯審查準確性、品牌語氣、SEO
4. **審批** — 利益相關者簽核
5. **排程** — 設置發布日期 / 時間
6. **發布** — 內容通過 API 上線

### 預覽 API

所有主要 Headless CMS 平台都支援草稿預覽：

- **Sanity**：使用 `useLiveQuery` 或 Presentation 工具進行實時預覽
- **Contentful**：使用單獨訪問令牌的預覽 API（`preview.contentful.com`）
- **Strapi**：帶 `status=draft` 查詢參數的草稿和發布系統（v5；取代 v4 的 `publicationState`）

在前端設置預覽路由（例如 `/api/preview`），進行身份驗證並渲染草稿內容。

### 角色和權限

| 角色 | 可以創建 | 可以編輯 | 可以發布 | 可以刪除 |
|------|:------:|:------:|:------:|:------:|
| 作者 | 是 | 自己的 | 否 | 自己的草稿 |
| 編輯 | 是 | 全部 | 是 | 草稿 |
| 管理員 | 是 | 全部 | 是 | 全部 |

## 平台比較

| 功能 | Sanity | Contentful | Strapi |
|------|--------|------------|--------|
| 託管 | 雲端（託管） | 雲端（託管） | 自託管或雲端 |
| 查詢語言 | GROQ | REST / GraphQL | REST / GraphQL |
| 免費套餐 | 慷慨 | 有限 | 開源（免費） |
| 實時協作 | 是（內建） | 有限 | 否 |
| 多語言支援 | 需要插件 | 原生支援 | 需要插件 |
| 最適合 | 開發者靈活性 | 企業多地區 | 預算 / 自託管 |
| 內容建模 | 代碼式 Schema | 網頁介面 | 網頁介面或代碼 |

### Sanity

**優點**：GROQ 查詢語言強大靈活。Schema 在代碼中定義（版本控制）。實時協作編輯。Portable Text 處理富文本。免費套餐慷慨。

**注意事項**：非開發者學習曲線較陡。Studio 自定義需要 React 知識。對 GROQ 查詢的供應商鎖定。

**香港行銷適用性**：最適合開發者和行銷人員緊密協作的情況。對內容豐富、模型複雜的網站效果好。如需雙語支援，需要仔細設計 Schema。

### Contentful

**優點**：成熟的企業平台。出色的多語言支援（適合中英文雙語）。強大的整合生態系統。有充分文檔的 API。

**注意事項**：定價隨內容類型和語言版本增加而增加。兩個單獨的 API（傳遞和管理）。較低套餐的速率限制可能緊張。

**香港行銷適用性**：最適合需要中英文雙語的企業。多語言 / 多地區需求時首選。對香港 + 大灣區內容策略特別有用。

### Strapi

**優點**：開源，可自託管。對數據的完全控制。無按席位定價。可自定義管理面板。插件生態系統。

**注意事項**：自託管意味著你負責基礎設施。生態系統比 Sanity / Contentful 小。從 V4 遷移到 V5 可能很重要。

**香港行銷適用性**：最適合有 DevOps 能力、想要完全控制且不希望有供應商鎖定的團隊。對預算有限的項目友好。

### 其他值得了解的平台

- **Hygraph** — GraphQL 原生，適合聯合和多源內容
- **Keystatic** — 基於 Git，適合開發者-內容混合工作流
- **Payload** — TypeScript 優先，自託管，代碼配置
- **Builder.io** — 帶 Headless 後端的視覺編輯器，適合非技術行銷人員
- **Prismic** — 基於 Slice 的內容建模，強大的 Next.js 整合

## 與香港行銷技能的整合

### 內容策略
CMS 內容模型執行一致的結構。定義與你的文案框架匹配的字段（標題、副標題、社會認同、CTA）。參見 **strategy-content-hk** 技能。

### 文案撰寫
CMS 確保內容格式一致。定義符合香港市場文案原則的字段。參見 **copywriting-hk** 技能。

### 發布策略
在 CMS 中加入「發布狀態」和「公告類型」字段，配合分階段發布計劃。參見 **strategy-launch-hk** 技能。

### 社交媒體內容
從 CMS 提取部落格內容，重新包裝為社交媒體貼文。確保 CMS 有 og:image 字段（對 WhatsApp 預覽重要）。參見 **social-content-hk** 技能。
